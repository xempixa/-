# AGENTS.md

## 1) 项目目标与边界
- **目标**：在不破坏现有功能的前提下，持续交付可维护、可验证、可回滚的增量改动。
- **边界**：
  - 仅实现明确需求范围内的功能与修复，不擅自扩展需求。
  - 不在无明确授权的情况下修改对外 API 语义、鉴权策略、数据模型核心约束。
  - 非紧急场景下不引入重型依赖或大规模架构重写。

## 2) 当前技术栈
> 若与实际不一致，以仓库内真实代码与锁文件为准；修改前先核对。

- **语言**：Python 3.x
- **后端框架**：FastAPI（Web API）
- **数据库访问**：SQLAlchemy（推荐 2.x 风格）
- **异步数据库会话**：`AsyncSession`
- **任务系统**：Worker / Scheduler（异步任务或定时任务）
- **运行环境**：本地开发 + 可部署环境（容器或进程托管）

## 3) 目录职责说明
> 以下为规范性职责边界；如仓库目录命名不同，请在不改变职责边界的前提下映射。

- `app/web`：HTTP 路由、请求/响应模型、鉴权入口、参数校验。
- `app/service`：业务编排层，承接 Web/Worker 调用，不直接耦合传输层细节。
- `app/repo` 或 `app/db`：数据访问层（CRUD、查询拼装、事务边界协作）。
- `app/models`：ORM 模型与领域模型。
- `app/worker`：异步任务执行器，只处理后台任务，不承载 HTTP 入口逻辑。
- `app/scheduler`：定时调度与触发，不直接承载复杂业务实现。
- `migrations`：数据库迁移脚本。
- `tests`：单元测试、集成测试、回归测试。
- `docs`：设计、运维、发布与验证文档。

## 4) 代码修改原则（小步、可运行、可回滚）
- **小步提交**：一次变更聚焦一个目标，避免“顺手大改”。
- **保持可运行**：每次改动后应至少通过最小可用验证（启动/核心测试/关键接口）。
- **可回滚**：
  - 避免不可逆的数据与接口破坏。
  - 变更应具备清晰回退路径（代码回退 + 配置回退 + 迁移回退策略）。
- **兼容优先**：优先新增而非替换；确需替换时保留过渡层。
- **可观测性**：关键路径补充必要日志与指标（不泄露敏感信息）。

## 5) 数据库修改原则（优先兼容，必要时补迁移）
- 优先采用**向后兼容**方式：
  - 新增列优先允许空值或提供安全默认值。
  - 先发布“读新写旧兼容态”，再逐步切换到“读新写新”。
- 涉及 schema 变更时，必须提供迁移脚本（`migrations`）。
- 禁止直接在生产环境手工改表替代迁移。
- 高风险 DDL（重命名/删列/改类型）需提供：
  - 风险评估
  - 数据迁移/回填方案
  - 回滚方案
- 索引与约束变更需评估对写入与查询性能的影响。

## 6) AsyncSession 使用规则
- **每个并发任务使用独立 `AsyncSession`**。
- **禁止跨协程/跨任务共享同一个 `AsyncSession` 实例**。
- 建议通过 `async_sessionmaker` 在请求作用域或任务作用域内创建并释放会话。
- 事务边界明确：
  - 需要原子性的操作放在同一事务中。
  - 读操作默认不长时间持有事务。
- 异常时正确回滚并关闭会话，避免连接泄漏。

## 7) Worker / Scheduler / Web 职责边界
- **Web**：处理同步请求-响应、参数校验、鉴权、快速返回。
- **Worker**：执行耗时/重试型后台任务；保证幂等与失败可恢复。
- **Scheduler**：负责任务触发策略（何时执行），不承担复杂业务逻辑。
- **边界约束**：
  - Web 不直接内嵌长耗时阻塞流程，应委托 Worker。
  - Scheduler 不直接写复杂业务，统一调用 service/worker 入口。
  - Worker 不暴露 HTTP 语义对象（如 Request/Response）到内部核心逻辑。

## 8) 安全规则
- 不伪造、猜测或硬编码私有接口参数；仅使用已声明、可审计的接口契约。
- 严禁将 `cookie`、`storage_state`、令牌、密钥、敏感日志写入前端可见区域或提交到仓库。
- `subprocess` 调用**禁止** `shell=True` 拼接命令；必须使用参数数组并做输入校验。
- 服务监听地址默认 `127.0.0.1`，仅在明确要求并完成安全评估后才开放到 `0.0.0.0`。
- 日志脱敏：对 token、密码、身份证号、手机号等敏感字段做掩码或不落盘。

## 9) 文档要求
以下文档需与代码变更保持同步：

1. `README`：项目简介、架构概览、核心能力、常见命令。
2. `QUICKSTART`：最小启动步骤（环境变量、依赖安装、启动与最小验证）。
3. `DEPLOY_LOCAL`：本地部署/联调说明（含数据库、队列、迁移、回滚）。
4. **验证步骤**：每次改动在 PR 或提交说明中提供可复现的验证命令与预期结果。

## 10) 提交输出格式
每次提交（或 PR 描述）至少包含以下三部分：

### 变更摘要
- 做了什么
- 为什么做
- 影响范围（模块/接口/数据库）

### 风险
- 潜在风险点
- 兼容性影响
- 回滚方案

### 验证方式
- 本地验证命令
- 测试范围（单测/集成/手工）
- 验证结果与关键截图/日志（如适用）
